Team guide · /prototype-optics
Prototype UIs that can't drift from Optics Three deterministic hooks block any save that uses an off-system color, size, class, or BEM structure. "Basically Optics" becomes provably Optics — no review needed to stay on-system.
lightbulb
The guarantee
A non-compliant file physically can't be written. The assistant gets the error and iterates until it complies — so the result is always on-system.

How it works

1
Scaffold Run /prototype-optics to drop tokens, the Optics bundle, and config into the project.
arrow_forward
2
Prototype Link vendor/optics.css and build with real Optics classes.
arrow_forward
3
Guard on save All three hooks run on every Write/Edit before the file lands.
arrow_forward
4
Pass / fix Clean → saved. Violation → blocked with a fix message → iterate.

The three guards

GuardBlocksRejected → fix
value raw colors; raw lengths (any property); wrong-category tokens; text that doesn't pair with its surface color: #f00 → var(--op-color-…); 12px → var(--op-space-small)
classname classes that aren't real Optics (typos, invented) unless prefixed or allow-listed btn--primry → btn--primary
BEM structure a block__element class used outside an element with the block class (HTML, on save) bare text-pair__title → wrap it in .text-pair
